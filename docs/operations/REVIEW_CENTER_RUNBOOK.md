# 导入审查中心 Runbook

适用范围：
- Phase 5 正式数据接入期间
- 管理员在导入审查中心处理 `station_name_mapping_proposals`
- 管理员在导入审查中心处理 `fault_import_review_queue`

入口：
- 页面：`/admin/review-center`
- 站名提议接口前缀：`/admin/station-name-proposals`
- 待确认队列接口前缀：`/admin/import-review-queue`

目标：
- 先把“站名映射”收敛稳定
- 再给待确认记录补站点归属
- 最后决定“导入 / 并单 / 驳回”

## 一、处理顺序

1. 先处理站名映射提议。
2. 再处理待确认队列里的站点指派。
3. 再决定单条记录是导入、并单还是驳回。
4. 最后抽样复核，并记录本批处理结论。

原因：
- 站名映射一旦稳定，后续同源记录可以批量套用，能显著减少人工成本。

## 二、进入前检查

进入审查中心前先确认：
- 当前导入批次对应的项目、来源系统、文件范围已明确。
- 本次处理责任人已明确。
- 已知站点名、别名、外部名称的映射规则可查询。
- 如需批量归属，先确认目标站点不是凭印象猜测，而是有证据支持。

推荐证据：
- 原始站名与现网站名高度一致。
- 原始内容里带有县区、位置、设备编号等可辅助判断的信息。
- 已存在同来源系统的稳定映射。

## 三、站名提议处理

处理对象：
- `station_name_mapping_proposals`

界面信息重点看：
- 外部站名
- 项目
- 来源系统
- 候选站点
- 置信度

### 3.1 单条通过

适用场景：
- 外部站名和候选站点关系明确
- 或者管理员能明确指定唯一正确站点

操作：
1. 在“站名映射提议”区域筛选 `pending`
2. 确认候选站点是否正确
3. 如候选站点为空，先在“批量归属站点”下拉框选择正确站点
4. 点击“通过”

结果：
- 写入 `station_external_names`
- 提议状态改为 `approved`

### 3.2 批量通过

适用场景：
- 多条记录来自同一来源系统
- 外部站名不同，但都能明确归到同一站点

操作：
1. 勾选多条提议
2. 如需要覆盖候选站点，先在“批量归属站点”选择目标站点
3. 点击“批量通过”

### 3.3 驳回

适用场景：
- 外部站名明显错误
- 当前证据不足，不能安全归属
- 候选站点不可信且暂无进一步线索

操作：
1. 单条点击“驳回”，或勾选后点击“批量驳回”

驳回后含义：
- 不写入正式映射表
- 不等于数据已导入
- 该提议后续若再次出现，应重新评估

## 四、待确认队列处理

处理对象：
- `fault_import_review_queue`

界面信息重点看：
- 原始站名/站点 token
- 项目
- 来源类型
- 当前指派站点
- 问题类型
- 原始描述

### 4.1 先做站点指派

原则：
- 未指派站点前，不做“导入”
- 批量动作优先于逐条动作

适用批量指派的场景：
- 同一批记录明显属于同一站点
- 同一来源系统、同一站名 token、同一项目
- 管理员已通过站名提议或原始内容确认归属

操作：
1. 在队列表中勾选记录
2. 在“批量指派站点”选择目标站点
3. 点击“批量指派站点”

结果：
- 只更新待确认记录的原始 payload 中的 `assigned_station_id`
- 不会直接导入故障

### 4.2 批量套用映射

适用场景：
- 当前选中的记录来自同一来源系统
- 其外部站名已经能够确定对应站点
- 需要顺手把正式映射也补齐，减少后续重复人工处理

操作：
1. 先完成站点指派
2. 点击“批量套用映射”

结果：
- 为对应外部站名写入 `station_external_names`
- 后续同来源记录能直接命中映射

注意：
- 批量套用映射前，必须确认这些外部站名确实应长期归属到同一站点

## 五、三种最终决策

### 5.1 导入

适用场景：
- 站点已明确
- 不是已有故障的重复项
- 原始记录应形成一条新的故障

操作：
1. 确认该条记录已有 `assigned_station_id`
2. 点击“导入”

结果：
- 生成新的 `fault_reports`
- 队列状态更新为已处理

### 5.2 并单

适用场景：
- 该记录只是已有故障的重复上报
- 不应再新建故障

操作：
1. 点击“并单”
2. 输入要并入的现有故障 ID
3. 提交

结果：
- 队列项状态更新为已处理
- `resolved_fault_id` 指向已有故障

并单前必须确认：
- 目标故障确实是同一事件，不只是同一站点
- 时间、描述、故障类型、设备线索不冲突

### 5.3 驳回

适用场景：
- 原始记录无效
- 数据缺失过多
- 无法判断真伪或归属，且当前批次不再继续追查

操作：
1. 单条点击“驳回”，或勾选后点击“批量驳回”

驳回后数据去向：
- 队列状态变为 `rejected`
- 不会写入正式故障表
- 原始 payload 保留，可审计、可回看

## 六、如何判断“可以批量归属”

满足以下条件时，优先批量处理：
- 同一项目
- 同一来源系统
- 相同或高度一致的外部站名
- 原始描述没有出现明显冲突的位置信息
- 候选站点或人工确认结果一致

以下情况不要批量归属：
- 同名但跨县区
- 原始描述出现多个可能站点
- 同一批记录里混有不同来源系统
- 设备编号、位置描述明显指向不同站点

## 七、抽样复核

每次批量操作后至少抽样复核：
- 批量通过的站名提议：抽 3 条
- 批量指派站点：抽 3 条
- 批量套用映射：抽 3 条
- 批量驳回：抽 3 条

复核项：
- 站点是否正确
- 项目是否正确
- 来源系统是否一致
- 记录状态是否符合预期
- 是否误伤其他项目或其他站点

## 八、过期机制

维护脚本：
- `python maintain_review_queue.py --database <db> --dry-run`
- `python maintain_review_queue.py --database <db> --apply --report <path>`

规则：
- `pending` 超过 30 天可被标记为 `expired`
- `expired` 代表本轮未处理完成，不代表数据已被正式导入或驳回

建议：
- 正式接入期间每天至少执行一次 `--dry-run`
- 每周至少执行一次带 `--report` 的维护

## 九、批次处理完成定义

一个批次可以认为“处理完成”，至少满足：
- 相关站名提议已清空或有明确驳回结论
- 待确认队列中不再有该批次的 `pending`
- 已处理记录已做抽样复核
- 处理结论已写入批次记录或交接记录

## 十、处理后的记录建议

建议记录以下信息：
- 批次 ID
- 处理人
- 处理时间
- 导入数 / 并单数 / 驳回数
- 新增映射数
- 抽样复核结果
- 遗留问题

## 十一、禁止事项

- 不要在未确认站点前直接导入
- 不要把“猜测最像”当作批量归属依据
- 不要把不同来源系统的记录混在一起批量套用映射
- 不要因为赶进度跳过抽样复核
- 不要直接改数据库绕过审查中心流程，除非进入专项数据修复并单独留痕
