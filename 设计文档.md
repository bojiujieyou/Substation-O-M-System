## 一、背景与目标

### 1.1 现状

- 当前存在三个独立系统：统一平台、智慧巡视、辅控系统。
- 变电站名称基本一致，但各系统设备独立、编号独立、故障类型和通知对象可能不同。
- 后续可能继续接入新项目，设计不能写死为 3 个项目。
- 现阶段样本数据尚未全部到位，必须先把模型、边界和导入策略设计清楚。

### 1.2 V1 目标

- 将多个项目整合到一个运维平台中。
- 保持项目间设备、故障类型、通知、权限相互隔离。
- 提供跨项目的统一运维视图，但不丢失项目归属。
- 为后续继续接入项目、补齐样本数据、扩展分析能力留出空间。

### 1.3 核心路线

V1 固定采用以下路线：

- 项目驱动
- 项目内独立槽位
- 故障类型可回溯
- 权限和通知按项目收口
- 历史连续性靠 `camera_slot_id` 串联

V1 明确不做：

- 跨项目物理摄像头统一建模
- 已关闭故障重开
- 复杂多级通知链
- 站点级权限控制

## 二、核心规则

### 2.1 项目标识

- `projects.code` 是永久外部标识。
- 对外 API、URL、导入参数统一使用 `project code`，不使用数据库主键。
- `code` 只允许新增，不允许修改。
- 如需“改名”，通过新增项目并停用旧项目实现，历史数据归属不变。

### 2.2 故障生命周期

- 故障状态机固定为三态：
  - `open`
  - `handling`
  - `closed`
- 允许的状态流转只有：
  - `open -> handling`
  - `open -> closed`
  - `handling -> closed`
- `closed` 不可逆。
- 故障复发时必须新建一条故障记录，不允许重开旧单。
- 槽位历史连续性通过 `camera_slot_id` 和时间线查询体现，不通过“单个故障反复开关”体现。

### 2.3 时间规则

- 数据库统一存储 UTC 时间。
- 前端统一按 `Asia/Shanghai` 展示。
- 导入时保留记录级原始时间审计字段：
  - `source_time_raw`
  - `source_timezone`

### 2.4 项目上下文

- 前端项目状态只认 URL：`?project=<code>`。
- 不使用 `sessionStorage` 记忆最近一次项目。
- URL 未带 `project` 时，按当前用户授权项目中 `sort_order` 最小者跳转。

### 2.5 项目停用

- 项目停用通过 `projects.is_active = 0` 实现。
- 停用后：
  - 历史数据仍可查询
  - 默认不出现在前台项目切换器
  - 不允许新增该项目数据
  - 用户授权关系保留，用于审计
  - 通知策略自动静默

### 3.3 `camera_slots`：逻辑位置槽位

用于回答“这个位置以前坏过没有”。

```sql
CREATE TABLE camera_slots (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    slot_code TEXT NOT NULL,
    station_id INTEGER NOT NULL,
    project_id INTEGER NOT NULL,
    location_desc TEXT NOT NULL DEFAULT '',
    area TEXT NOT NULL DEFAULT '',
    channel_number INTEGER,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(station_id, project_id, slot_code),
    UNIQUE(station_id, project_id, location_desc, area, channel_number),
    FOREIGN KEY (station_id) REFERENCES stations(id),
    FOREIGN KEY (project_id) REFERENCES projects(id)
);
```

规则：

- `slot_code` 是槽位的稳定业务标识，V1 为必填。
- 历史迁移时按 `project_code + station_key + channel_number + normalized_location` 生成默认编码。
- 手工新增时必须填写，或由系统生成临时编码。
- V1 槽位仅在项目内独立，不做跨项目物理位置统一建模。

### 3.4 `cameras`：物理设备实例

```sql
CREATE TABLE cameras (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    slot_id INTEGER,
    station_id INTEGER NOT NULL,
    project_id INTEGER NOT NULL,
    project_camera_code TEXT,
    camera_index TEXT,
    area TEXT,
    location_desc TEXT,
    ip_address TEXT,
    channel_port INTEGER,
    channel_number INTEGER,
    status TEXT DEFAULT 'active',
    replaced_by_camera_id INTEGER,
    retired_at TIMESTAMP,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (slot_id) REFERENCES camera_slots(id),
    FOREIGN KEY (station_id) REFERENCES stations(id),
    FOREIGN KEY (project_id) REFERENCES projects(id)
);
```

部分唯一索引：

```sql
CREATE UNIQUE INDEX idx_cameras_one_active_per_slot
ON cameras(slot_id)
WHERE status = 'active';
```

规则：

- 同一槽位允许多条 `replaced` / `retired` 历史设备。
- 同一槽位只能有一条 `active` 设备。
- 当前设备通过 `status='active'` 查询得到，不在 `camera_slots` 中保存缓存字段。

建议提供只读视图供查询使用：

```sql
CREATE VIEW v_camera_slots_with_current_camera AS
SELECT
    s.id AS slot_id,
    s.slot_code,
    s.station_id,
    s.project_id,
    s.location_desc,
    s.area,
    s.channel_number,
    c.id AS current_camera_id,
    c.project_camera_code,
    c.ip_address,
    c.camera_index
FROM camera_slots s
LEFT JOIN cameras c
    ON c.slot_id = s.id
   AND c.status = 'active';
```

补充说明：

- `status` 和 `closed_at` 已存在于当前基线 schema 中，迁移时不得盲目重复 `ADD COLUMN`。
- 所有 `ALTER TABLE` 必须先执行字段存在性检查：
  - 不存在则新增
  - 已存在且语义一致则跳过
  - 已存在但语义不一致则进入单独迁移分支处理
- 现有 `system_type` 列保留，只作遗留审计字段。
- `status` 的合法值固定为 `open` / `handling` / `closed`。
- `assigned_to` 用于记录当前处理责任人，支持 `handling` 状态的责任归属与 SLA 统计解释。
- 页面展示故障类型使用 `fault_type_label_snapshot`。
- 统计优先按 `semantic_group`，缺失时按 `fault_type_code` 聚合。
- `tags_json` 为 V2 预留字段，V1 仅存储和查看，不做正式统计。

幂等字段关系：

- `idempotency_key`
  - 继续保留，用于现有手工报修/API 提交流程的短窗口幂等控制
  - 主要服务在线提交场景
- `source_record_key`
  - 新增，用于历史导入、批量导入、外部同步的记录级幂等识别
  - 主要服务离线/外部来源数据
- 两者不是同义字段，不直接互相替代。
- Phase 1 迁移时：
  - 保留现有 `idempotency_key` 原值
  - 仅在能够确认历史记录来源语义时，才允许按规则回填 `source_record_key`
  - 无法确认时，`source_record_key` 允许为空，避免把旧键错误迁移成新语义

历史处理人字段迁移路径：

- 当前已有 `handler_name` 与 `handler_note` 文本不得丢失。
- 迁移后两字段继续保留，作为历史审计和可读展示字段。
- `assigned_to` 是新增责任归属字段，语义不同于 `handler_name`，不做盲目覆盖。
- Phase 1 迁移时按以下规则处理：
  1. 保留原始 `handler_name`、`handler_note`
  2. 若 `handler_name` 能与 `users.username` 做唯一精确匹配，则允许回填 `assigned_to`
  3. 若无法唯一匹配，则 `assigned_to = NULL`
- V1 前台和管理员详情页仍需能查看历史 `handler_name` 与 `handler_note`。

历史 `fault_type` 迁移路径：

- 当前已有 `fault_type` 文本不得丢失。
- 迁移时按以下顺序处理：
  1. 原始 `fault_type` 文本先完整写入 `fault_type_label_snapshot`
  2. 若能通过映射表或当前版本字典确认唯一 `fault_type_code`，则同时写入：
     - `fault_type_code`
     - `fault_type_version_id`
  3. 若无法安全映射，则保留：
     - `fault_type_label_snapshot`
     - `fault_type_code = NULL`
     - `fault_type_version_id = NULL`

新发现的外部站名提议不复用故障待确认队列，避免“站名提议”和“故障导入问题”混在同一处理语义中。

```sql

    id INTEGER PRIMARY KEY AUTOINCREMENT,
    import_batch_id INTEGER,
    project_id INTEGER,
    source_system TEXT NOT NULL,
    external_name TEXT NOT NULL,

    confidence_score REAL,
    raw_context_json TEXT,
    status TEXT NOT NULL DEFAULT 'pending',  -- pending / approved / rejected / expired
    reviewer_id INTEGER,
    reviewed_at TIMESTAMP,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (import_batch_id) REFERENCES import_batches(id),
    FOREIGN KEY (project_id) REFERENCES projects(id)
);
```

规则：

- 导入发现新的外部站名时，先写入 `station_name_mapping_proposals`
- 管理员确认后，再写入 `station_external_names`
- 不允许导入流程绕过提议表直接写入正式映射表
- 超过 30 天未处理的提议可标记为 `expired`

`confidence_score` 规则：

- 取值范围固定为 `0.0 ~ 1.0`
- V1 使用确定性启发式评分，不引入复杂模型
- 建议规则：
  - `1.0`：原始名称与 canonical name 完全一致
  - `0.9`：归一化后精确命中 `station_external_names` 或 `station_aliases`
  - `0.7`：受限模糊匹配后仅命中一个候选站点
  - `0.4 ~ 0.6`：依赖县区 + 部分名称相似度的弱匹配
  - `NULL`：无法形成可信候选或候选不唯一
- 管理后台展示时：
  - `>= 0.9` 视为高置信度
  - `>= 0.7 且 < 0.9` 视为中置信度
  - `< 0.7` 或 `NULL` 视为低置信度，需要人工谨慎处理

## 四、关键流程

### 4.1 设备替换

设备替换事务顺序固定为：

1. 校验旧设备属于目标槽位且状态为 `active`
2. 插入新设备为 `active`
3. 旧设备更新为 `replaced` 并写入 `replaced_by_camera_id`
4. 提交事务

说明：

- 当前设备不写缓存字段，因此不存在 `current_camera_id` 悬空问题。
- 槽位历史通过设备实例链和故障记录链查询。

### 4.2 故障状态流转

规则：

- 新建故障默认进入 `open`
- `open -> handling` 时写入 `handling_started_at`
- `open -> handling` 时写入 `assigned_to`
- `handling -> closed` 时写入 `closed_at`
- `open -> closed` 时仅写入 `closed_at`
- `closed` 不可逆

通知触发：

- 创建通知：故障创建并进入 `open`
- 升级通知：超过阈值仍未 `closed`
- 关闭通知：进入 `closed`

### 4.3 故障 KPI 口径

- 响应时长 = `handling_started_at - created_at`
- 关闭时长 = `closed_at - created_at`
- 对于直接 `open -> closed`：
  - 响应时长记为 `NULL`
  - 关闭时长正常计算

V1 KPI 受众：

- 运维人员
- 管理员

V1 不承诺：

- 面向甲方或监管的大型报表
- 复杂 SLA 体系

### 4.4 历史故障导入

导入流程固定为：

1. 读取源数据
2. 解析时间并归一到 UTC
3. 生成 `source_record_key` 或候选键
4. 做站点、槽位、设备、故障类型映射
5. 可安全入库则写 `fault_reports`
6. 不可安全入库则写 `fault_import_review_queue`
7. 同时输出报告文件

`source_record_key` 统一规则：

```text
project_code:source_type:sha256(raw_external_id_or_canonical_row)
```

优先级：

1. 使用源系统原始唯一 ID
2. 无唯一 ID 时，使用规范化行内容生成候选键
3. 若无法证明“完全相同内容只能是一条故障”，则不自动入库，进入待确认队列

规则：

- 所有导入器必须调用同一个共享模块生成 `source_record_key`
- 不允许各导入器自行实现

### 4.5 站名映射提议流

维护机制：

1. 迁移和数据探查阶段初始化一批映射
2. 导入发现新的外部名称时，写入 `station_name_mapping_proposals`
3. 管理员确认提议后写入 `station_external_names`
4. 未确认前不进入正式映射表

### 4.6 通知停用与项目下线

- 通知策略和通知目标默认采用“停用优先”，不鼓励硬删除。
- 管理后台必须提供启用/停用入口。
- 项目停用后，通知自动静默。

- `<= 5%`

  - 依据：人工清洗成本约 1-2 人天，可在单迭代内消化
- `> 5% 且 <= 15%`

- `>= 90%`

- `>= 75% 且 < 90%`
  - 允许自动导入 + 待确认队列双轨
- `< 75%`
  - 阻断全量接入

#### 时间戳可解析率

- `>= 95%`
  - 才允许进入正式统计与幂等流程
- `< 95%`
  - 阻断进入正式统计逻辑

## 八、迁移、回滚与实现约束

### 8.0 变更控制规则

- 本文档状态为 `FROZEN`，表示 V1 功能边界已冻结。
- V1 冻结期间仅允许两类修改：
  - 勘误：修正 schema 漏字段、接口遗漏、文字错误、章节不一致
  - 澄清：补充不改变功能边界的解释性说明
- 任何功能扩展、模型扩展、流程新增都必须：
  - 提升版本号
  - 记录触发原因和日期
  - 明确影响章节
- 默认解冻权限归属于方案负责人或明确授权的评审人，不允许无痕修改冻结版正文。

### 8.1 统一连接工厂

所有数据库连接必须通过统一连接工厂创建，工厂内强制执行：

```sql
PRAGMA foreign_keys = ON;
PRAGMA busy_timeout = 30000;
```

适用范围：

- Web 请求连接
- 迁移脚本连接
- CLI 导入脚本连接
- 后台任务连接

### 8.2 迁移与导入脚本标准能力

所有迁移与批量导入脚本统一支持：

- `--dry-run`
- `--report <path>`
- `--fail-on <rule>`
- `--timezone-default Asia/Shanghai`

迁移脚本实现原则：

- 不允许假设文档中的所有 `ALTER TABLE` 都能直接执行。
- 所有迁移必须先对照 3.0 节的 schema 基线审计结果分类处理：
  - 可直接 `ALTER TABLE` 的字段
  - 需要字段存在性检查的字段
  - 必须重建表的对象

当前明确需要重建或高概率需要重建的表：

- `cameras`

原因：

- 需要新增 `slot_id`
- 需要新增 `project_id`
- 需要新增 `status`
- 需要新增 `replaced_by_camera_id`
- 需要新增 `retired_at`
- 需要建立“同槽位仅一条 `active`”的部分唯一索引
- 其中 `project_id` 的语义要求与现有数据初始化过程不适合简单 `ALTER + NOT NULL`

推荐重建步骤：

1. 创建新表 `cameras_v2`
2. 按基线数据复制旧数据并补齐：
   - `project_id`
   - `slot_id`
   - `status='active'`
   - 显式保留原 `id`
3. 建立新索引与约束
4. 做数据校验
5. 切换表名
6. 保留回滚前备份

关键约束：

- 重建 `cameras` 时不得依赖新的自增值。
- 复制到 `cameras_v2` 时必须显式插入旧 `id`，确保现有 `fault_reports.camera_id` 引用不断链。
- 若无法保留原 `id`，则必须同步提供 `fault_reports.camera_id` 全量映射修复方案；V1 默认不采用该路径。

风险说明：

- `cameras` 重建是 Phase 1 中停机风险和校验成本最高的一步
- 该步骤应单独列入迁移检查清单，不与普通 `ALTER TABLE` 混为同一类操作

对于待确认队列和站名提议的过期机制，V1 固定采用部署侧定时维护任务，不使用应用内调度器或页面惰性触发。

建议：

- 每日由系统计划任务、cron 或任务计划程序执行一次维护脚本
- 脚本负责：
  - 将超期 `fault_import_review_queue.pending` 标记为 `expired`
  - 将超期 `station_name_mapping_proposals.pending` 标记为 `expired`
  - 输出维护报告

### 8.3 正式回滚策略

- 正式迁移失败后的主回滚路径是“备份恢复”。
- V1 不承诺通用脚本级 `--rollback`。
- 实施前必须包含：
  - 数据库强制备份
  - 备份文件校验
  - 恢复演练步骤

### 8.4 批次错误的应急处理

- 当某批导入成功后又发现系统性错误时，V1 以 `import_batch_id` 作为审计锚点。
- 默认应对方式：
  - 导出受影响记录
  - 走批次撤销脚本或数据库恢复流程
- V1 不承诺通用 UI 一键回滚。

## 十一、风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| 样本数据结构差异大 | 导入规则返工 | 先完成 Data Discovery Sprint，再全量导入 |
| `slot_code` 生成冲突率高 | 槽位错链 | 以阈值阻断，先补规则再导入 |
| 时间戳格式混乱 | 统计与幂等失真 | 记录级保存原始时间与时区，统一转 UTC |
| 站名“基本一致”但不完全一致 | 匹配错误 | 用 `station_external_names` 承接，不依赖脚本猜测 |
| 队列积压 | 人工处理失控 | 提供分组、批量处理和过期归档 |
| 迁移后发现整批导入错误 | 数据污染 | 以 `import_batch_id` 审计，走批次撤销脚本或备份恢复 |
| SQLite 外键未开启 | 孤儿数据 | 统一连接工厂强制 `PRAGMA foreign_keys = ON` |

## 十三、结论

本版为 V1 冻结版设计方案，核心决策如下：

- 项目使用永久 `code`
- V1 采用项目内独立槽位
- 故障三态流转，`closed` 不可逆
- 复发新建，不重开
- 故障类型版本化，保留快照与语义组
- 待确认队列为正式表
- 站名映射通过结构化表和提议流维护
- 时间统一入库 UTC
- 项目上下文只认 URL
- 正式迁移失败以备份恢复为主

## 十四、关键决策记录

本章用于记录已生效的关键取舍，回答“为什么这样定、放弃了什么、后续何时重看”。  
状态说明：

- `accepted`：已采纳并生效
- `revisit`：后续版本可重看

### ADR-001：项目外部标识统一使用 `projects.code`

- 决策：对外接口、URL、导入参数统一使用 `projects.code`，不使用数据库主键。
- 日期：2026-04-02
- 状态：accepted
- 背景：项目数量会扩展，数据库主键顺序不稳定，不能依赖“1=统一平台”这类隐式约定。
- 备选方案：
  - 使用数据库 `id`
  - 前台和脚本混用 `id` 与 `code`
- 原因：`code` 可读、稳定、适合 URL 和导入参数，也避免迁移时种子顺序变化带来的问题。
- 影响：
  - 所有对外 API 都以 `code` 为准
  - 迁移回填必须通过 `code` 查询项目 id
- 何时重看：仅当未来需要对接外部主数据平台并统一项目编码体系时重看

### ADR-002：V1 不做跨项目物理摄像头统一建模

- 决策：V1 采用项目内独立槽位模型，不做跨项目物理摄像头去重。
- 日期：2026-04-02
- 状态：accepted
- 背景：同一物理摄像头可能被多个项目引用，但样本数据尚未充分验证，直接统一建模会显著增加复杂度。
- 备选方案：
  - 引入独立的物理资产主表
  - 在 V1 就建立跨项目槽位关联
- 原因：先按 `project_id` 隔离最稳，能满足当前项目收口需求，也不会阻断后续扩展。
- 影响：
  - 同一物理位置在不同项目下可存在不同槽位
  - 后续如需跨项目分析，可在 V2 引入弱关联键
- 何时重看：当明确需要跨项目物理资产分析或统一维护时

### ADR-003：`camera_slots` 不保存当前设备缓存

- 决策：删除 `current_camera_id`，当前设备统一通过 `cameras.status='active'` 查询。
- 日期：2026-04-02
- 状态：accepted
- 背景：缓存字段会带来双写一致性问题，替换流程中最容易出现悬空或指错设备。
- 备选方案：
  - 在 `camera_slots` 中保留当前设备缓存字段
  - 同时保留缓存和视图
- 原因：一次 JOIN 的成本小于数据错链的成本，SQLite 在当前规模下完全可承受。
- 影响：
  - 设备替换事务更简单
  - 可通过只读视图统一封装查询入口
- 何时重看：当目标规模下槽位查询 P95 持续超过阈值时

### ADR-004：故障采用三态状态机，`closed` 不可逆

- 决策：故障状态固定为 `open -> handling -> closed`，`closed` 不可逆。
- 日期：2026-04-02
- 状态：accepted
- 背景：故障通知、SLA、归档和处理链路都依赖稳定的状态机。
- 备选方案：
  - 两态：`open / closed`
  - 四态或更复杂状态机
  - 支持 `closed -> open` 重开
- 原因：三态与现有实现一致，足够支撑运维闭环，又不引入复杂状态爆炸。
- 影响：
  - 复发故障必须新建
  - 响应时长和关闭时长可明确分开计算
- 何时重看：当业务明确提出待确认、挂起、转派等复杂流转需求时

### ADR-005：故障复发新建，不重开旧单

- 决策：故障复发时创建新记录，不允许重开已关闭工单。
- 日期：2026-04-02
- 状态：accepted
- 背景：如果支持重开，单条故障会跨越多次关闭/复发，审计、通知和统计都会变复杂。
- 备选方案：
  - 允许 `closed -> open`
  - 按故障类型决定是否允许重开
- 原因：故障单表示一次事件，槽位时间线才表示连续历史。
- 影响：
  - 历史连续性靠 `camera_slot_id`
  - 单条故障生命周期保持简单清晰
- 何时重看：当明确要求把单个故障做成长生命周期工单时

### ADR-006：故障类型采用版本化 + 快照 + `semantic_group`

- 决策：故障类型采用版本表，故障记录保存快照，并通过 `semantic_group` 标记跨版本语义连续性。
- 日期：2026-04-02
- 状态：accepted
- 背景：项目故障类型会调整，历史故障必须保留“当时怎么定义”的含义。
- 备选方案：
  - 只保存纯文本快照
  - 只保存当前字典，不做版本化
- 原因：版本化保证历史不变义，`semantic_group` 为未来统计分析提供稳定语义维度。
- 影响：
  - 发布新版本时必须做 diff 审核
  - 历史统计优先按 `semantic_group`
- 何时重看：当 V2 需要更复杂的语义谱系管理时

### ADR-007：待确认队列是正式业务表，不是临时文件

- 决策：所有无法安全入库的记录进入 `fault_import_review_queue`，CSV 仅作为导出视图。
- 日期：2026-04-02
- 状态：accepted
- 背景：历史导入和多源数据接入一定会遇到无法自动判定的记录，必须有系统内正式处理流。
- 备选方案：
  - 只输出 CSV 报告
  - 把待确认内容塞进批次 JSON 字段
- 原因：正式表可追踪、可批量处理、可审计，也便于后续补录和复核。
- 影响：
  - 管理后台需要提供队列页面和批量操作
  - 待确认项可按状态、问题类型和站点分组
- 何时重看：当 V2 需要自动规则引擎批量消化队列时

### ADR-008：站名归一化通过 `station_external_names` + 提议流维护

- 决策：导入和迁移优先查结构化映射表 `station_external_names`，新外部站名走提议确认流，不自动直写正式表。
- 日期：2026-04-02
- 状态：accepted
- 背景：“变电站名称基本一致”并不等于完全一致，脚本级模糊匹配不可长期依赖。
- 备选方案：
  - 继续靠模糊匹配
  - 导入成功后自动写入正式映射表
- 原因：结构化映射可审计、可追溯，也能避免错误映射沉淀成正式数据。
- 影响：
  - Data Discovery Sprint 必须先做站名规范化清查
  - 管理后台需支持映射提议确认
- 何时重看：当未来接入统一外部主数据服务时

### ADR-009：`source_record_key` 必须统一生成

- 决策：所有导入器必须调用同一个共享模块生成 `source_record_key`。
- 日期：2026-04-02
- 状态：accepted
- 背景：不同导入器如果各自实现，会导致同一条源数据生成不同幂等键，破坏去重。
- 备选方案：
  - 各导入器自行实现
  - 只靠数据库唯一索引兜底
- 原因：统一生成逻辑比“大家记得保持一致”更可靠。
- 影响：
  - 导入器不得自行发明哈希规则
  - 幂等逻辑可复用、可测试
- 何时重看：当未来需要跨系统同步并统一外部事件主键时

### ADR-010：数据库连接统一强制开启外键约束

- 决策：所有数据库连接都必须通过统一连接工厂创建，并强制执行 `PRAGMA foreign_keys = ON`。
- 日期：2026-04-02
- 状态：accepted
- 背景：SQLite 默认不强制外键，如果连接入口分散，极易产生孤儿数据。
- 备选方案：
  - 在个别脚本里手写 PRAGMA
  - 只在文档里要求
- 原因：这类规则必须固化在代码入口层，而不是依赖记忆。
- 影响：
  - Web、迁移、CLI、后台任务都要统一走连接工厂
  - 通知和权限等外键关系才能真正可靠
- 何时重看：仅当未来数据库迁移到原生强外键数据库时

### ADR-011：前端项目上下文只认 URL

- 决策：项目上下文只认 `?project=<code>`，不使用 `sessionStorage` 记忆。
- 日期：2026-04-02
- 状态：accepted
- 背景：多 Tab 场景下，“记忆上次项目”容易污染当前页面行为。
- 备选方案：
  - `sessionStorage` 恢复最近项目
  - 强制进入项目选择页
- 原因：URL 是最稳定、可分享、可刷新、可回溯的状态来源。
- 影响：
  - 无参数时统一按授权排序首项跳转
  - 多 Tab 不互相干扰
- 何时重看：当未来引入更复杂的工作区或多项目并行视图时

### ADR-012：项目停用后历史可查、通知静默

- 决策：项目停用后保留历史可查询性，默认不展示，不允许新增，通知自动静默。
- 日期：2026-04-02
- 状态：accepted
- 背景：项目下线是现实场景，必须避免“停用即消失”带来的审计断链。
- 备选方案：
  - 完全隐藏停用项目
  - 停用后仍像正常项目一样展示和通知
- 原因：历史数据必须保留，但新业务和通知需要立即收口。
- 影响：
  - 权限关系保留作审计
  - 前台默认不展示停用项目
- 何时重看：当未来需要归档库或冷热分层时

### ADR-013：正式迁移失败后的主回滚路径是备份恢复

- 决策：V1 正式迁移失败后，以数据库备份恢复为主回滚路径，不承诺通用脚本级 rollback。
- 日期：2026-04-02
- 状态：accepted
- 背景：SQLite 下跨多步迁移做“可靠反向回滚”复杂且脆弱，容易产生假回滚。
- 备选方案：
  - 每个迁移脚本都实现 `--rollback`
  - 关键脚本支持 rollback，其余靠备份
- 原因：备份恢复简单、可验证、操作风险更低。
- 影响：
  - 实施前必须备份、校验并准备恢复步骤
  - 批量错误处理以 `import_batch_id` 和恢复流程为主
- 何时重看：当未来迁移到支持更强事务迁移能力的数据库时

### ADR-014：V1 继续使用 SQLite

- 决策：V1 数据库继续使用 SQLite，不在本阶段切换到 MySQL、PostgreSQL 等服务型数据库。
- 日期：2026-04-02
- 状态：accepted
- 背景：当前部署环境是本地开发 + NAS，运维和发布成本要求低，现有系统也已经建立在 SQLite 之上。
- 备选方案：
  - 迁移到 PostgreSQL
  - 迁移到 MySQL
  - V1 同时维护双数据库适配
- 原因：SQLite 对当前规模和部署方式最友好，能以最低迁移成本支撑 V1 落地。
- 影响：
  - 必须显式开启外键约束
  - 回滚策略以备份恢复为主
  - 视图、索引和迁移策略需考虑 SQLite 特性
- 何时重看：当并发、数据量、分析需求或运维环境明显超出 SQLite 适用范围时
