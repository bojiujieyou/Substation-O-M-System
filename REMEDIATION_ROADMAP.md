# 变电站图像监控运维平台整改路线图 v2

更新时间：2026-04-24

## 目标

这份文档替代旧版“25 项一次性整改清单”。

本版只保留三类内容：

1. 当前仍然存在、需要执行的风险项
2. 已经完成但需要复核的修复项
3. 不属于安全阻断、应拆分为独立工程的结构性优化项

本文档已去除真实凭据。任何审计、复盘、路线图文档都不得再记录可用密钥、连接串或 Token。

---

## 结论

原审计方向基本正确，但旧版路线图存在四个问题：

1. 文档本身包含真实凭据，形成二次泄露
2. 多个问题已经修复，状态未更新，继续作为 P0 会误导执行
3. 部分措施不可直接落地，例如一次性上线严格 CSP
4. 大量“代码结构重构”混入安全整改主线，导致范围失控

因此执行上改为两条主线：

- 主线 A：发布阻断与认证安全
- 主线 B：部署可靠性与运行稳定性

大重构、模板拆分、包结构重组、数据库大迁移从本路线图剥离，进入单独 backlog。

## 主线 A：发布阻断与认证安全

### A0. 文档去敏与凭据轮换

状态：`未开始`

风险：

- 工作目录或审计材料中出现真实连接串、API Key、Token
- 即使代码已修，历史泄露仍然有效

执行项：

- [ ] 删除所有文档中的真实凭据，仅保留脱敏示例
- [ ] 轮换 PostgreSQL 用户密码
- [ ] 轮换 NVIDIA API Key
- [ ] 轮换 `API_TOKEN`
- [ ] 清点部署环境、CI、个人开发机是否仍保留旧值
- [ ] 增加启动校验：生产环境下 `DATABASE_URL`、`API_TOKEN`、`SECRET_KEY` 缺失或为默认值时拒绝启动

完成标准：

- 所有文档和脚本中不再出现可用密钥
- 新凭据已完成替换
- 旧凭据已确认失效

### A1. 未登录访问面收口

状态：`部分完成`

当前判断：

- 统计导出和统计数据接口已不再公开
- 设计变体页仍公开，且不应属于生产可匿名访问面

执行项：

- [ ] 从未登录白名单中移除 `design_variant_index`
- [ ] 从未登录白名单中移除 `design_variant_statistics`
- [ ] 明确未登录仅允许：`/`、`/statistics`、`/login`
- [ ] 增加回归测试，覆盖页面路由与 API 路由的匿名访问矩阵

完成标准：

- 未登录仅能访问首页、统计页、登录页
- 其余页面跳转登录，其余 API 返回 401/403

### A2. 管理员初始化收口

状态：`部分完成`

当前判断：

- `init_admin.py` 已要求显式提供 `INIT_ADMIN_PASSWORD`
- 但容器启动脚本对未设置密码采用“跳过管理员初始化”的宽松策略

执行项：

- [ ] 区分两类场景：
  - 首次初始化：必须提供 `INIT_ADMIN_PASSWORD`
  - 已有管理员：允许跳过
- [ ] 在 `docker-entrypoint.sh` 中显式判断管理员是否已存在
- [ ] 如果数据库为空且未设置初始化密码，则直接失败退出
- [ ] `.env.example` 保留注释示例，不再提供任何默认口令

完成标准：

- 首次部署不会因为遗漏环境变量而进入“无管理员但服务启动”的模糊状态
- 不存在默认口令，也不存在静默跳过初始化的首发风险

### A3. 密码哈希升级

状态：`未开始`

当前判断：

- 仍在使用单次 SHA-256 + salt
- 这不适合作为生产密码存储方案

执行项：

- [ ] 选型 `bcrypt` 或 `argon2`
- [ ] 统一 `auth.py` 与 `init_admin.py` 的哈希实现
- [ ] 实现惰性迁移：
  - 旧 SHA-256 校验成功后
  - 立即重写为新哈希格式
- [ ] 补充回归测试

完成标准：

- 新建与登录后迁移的账户均使用强哈希
- 不要求一次性批量重置全部用户密码

### A4. Session 与基础认证配置

状态：`未开始`

当前判断：

- `SECRET_KEY` 缺失时仍会随机生成，导致重启后 session 全部失效
- `SESSION_COOKIE_SECURE` 仍由环境变量自由控制，生产未强制
- 未设置明确会话寿命

执行项：

- [ ] 生产环境 `SECRET_KEY` 缺失或使用示例值时拒绝启动
- [ ] 增加 `PERMANENT_SESSION_LIFETIME`
- [ ] 生产环境强制 `SESSION_COOKIE_SECURE=True`
- [ ] 保留 `SESSION_COOKIE_HTTPONLY=True`
- [ ] 继续使用 `SameSite=Lax`，如后续接入跨站能力再重议

完成标准：

- 生产环境不会因配置疏漏进入“弱 session 配置”
- 重启不会无提示失效全部会话

### A5. 登录限流

状态：`未开始`

旧版问题：

- 把限流依赖于 Gunicorn 多 worker 改造，这个依赖过重

修正后的执行原则：

- 限流应先独立落地
- 只要后端状态存储可共享，就不需要等部署形态切换

执行项：

- [ ] 引入 `Flask-Limiter` 或等效方案
- [ ] 存储优先级：
  - Redis
  - 数据库/文件共享存储
  - 进程内内存仅作开发降级
- [ ] 登录接口默认阈值维持 `5 次 / 5 分钟`
- [ ] 针对用户名和 IP 组合做限流，避免纯 IP 误伤过大

完成标准：

- 多进程和单进程下限流行为一致
- 暴力尝试不会因 worker 隔离而绕过

### A6. CSRF 保护

状态：`未开始`

当前判断：

- 当前配置为全局关闭
- 旧版说明“JSON API 不受 CSRF 影响”不够严谨

执行原则：

- 这项仍应做，但不能在现有模板和前端 fetch 未统一前直接硬开

执行项：

- [ ] 先梳理所有有状态写操作端点
- [ ] 在基础模板中输出 CSRF token
- [ ] 封装统一 fetch helper，自动带 `X-CSRFToken`
- [ ] 分批放开校验，不做一次性全站开关
- [ ] 对 webhook / 非浏览器调用端点明确豁免清单

完成标准：

- 所有浏览器发起的写操作请求都带上并校验 CSRF token
- 不引入大面积前端回归

### A7. 安全响应头

状态：`未开始`

旧版问题：

- 旧版直接提议 `Content-Security-Policy: script-src 'self'`
- 当前模板仍大量依赖内联 `<script>` 和 `onclick`
- 直接上线严格 CSP 会导致现网页面失效

执行项：

- [ ] 立即补充低风险响应头：
  - `X-Frame-Options: DENY`
  - `X-Content-Type-Options: nosniff`
  - `Referrer-Policy: strict-origin-when-cross-origin`
  - `Permissions-Policy: camera=(), microphone=(), geolocation=()`
- [ ] CSP 分两步做：
  - 第一步：先 inventory 内联脚本和内联事件
  - 第二步：改为 nonce/hash 或外链脚本后，再上严格 CSP
- [ ] 如需先落 CSP，采用 `Report-Only` 模式观测，而不是直接阻断

完成标准：

- 通用安全响应头已上线
- CSP 有独立迁移计划，不直接造成页面故障

## 从安全主线拆出的事项

以下事项保留价值，但不再作为发布阻断：

### C1. 应用结构重构

- `create_app()` 工厂化
- 蓝图拆分
- scoped/legacy 去重
- 包结构重组

建议：

- 单独立项
- 以“降低维护成本”为目标，不和安全修复混排
- 采用小步迁移，不做一次性大爆炸

### C2. 数据库大迁移

- 主键统一 `BIGINT`
- 全量 `TIMESTAMPTZ`
- 外键策略系统化

建议：

- 单独输出“数据库演进方案”
- 先回答三个问题再做：
  1. 当前数据规模是否已经逼近 `INTEGER` 风险
  2. 现有时间字段是否已造成明确业务错误
  3. 外键删除策略是否已有实际事故

如果没有明确压力，不应插入当前安全主线。

### C3. 性能与前端体积治理

- 统计查询 CTE 化
- 游标分页
- 照片查询限制
- 模板拆分

建议：

- 先基于真实慢查询或接口耗时数据排优先级
- 不要把“可优化”误写成“必须前置”
